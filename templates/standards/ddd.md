# DDD 领域驱动设计编码约束

本文件定义项目遵循 DDD（Domain-Driven Design）方法论的核心约束。

---

## 一、战略设计

### 1.1 限界上下文（Bounded Context）

- 每个子域对应一个独立的限界上下文
- 限界上下文之间通过接口或事件通信，禁止直接共享数据库表
- 上下文边界是物理边界（独立模块/包），不仅是逻辑分区
- 同一概念在不同上下文中可以有不同命名和模型

### 1.2 上下文映射（Context Map）

- 明确上下文之间的关系：共享内核（Shared Kernel）、防腐层（ACL）、开放主机服务（OHS）等
- 跨上下文调用必须通过应用服务接口，禁止跨上下文直接调用领域逻辑
- 外部系统的集成通过防腐层（Anti-Corruption Layer）隔离

### 1.3 子域分类

- **核心域（Core Domain）**：业务核心竞争力，投入最优质资源，严格 DDD 建模
- **支撑域（Supporting Domain）**：辅助核心业务，可适当简化
- **通用域（Generic Domain）**：通用功能（认证、通知），优先使用成熟方案

## 二、战术设计

### 2.1 聚合根（Aggregate Root）

- 每个聚合有且仅有一个聚合根实体
- 聚合根负责维护聚合内的一致性不变量
- 聚合之间的引用只通过聚合根 ID，禁止持有其他聚合根的对象引用
- 聚合要尽量小，一个事务只修改一个聚合

### 2.2 实体（Entity）

- 有唯一标识符，标识符不随属性变化而改变
- 通过标识符判断相等性，而非属性比较
- 实体的生命周期限定在其所属聚合内

### 2.3 值对象（Value Object）

- 无唯一标识，通过属性值判断相等性
- 不可变（immutable），修改时返回新实例
- 优先使用值对象建模概念，能用值对象就不用实体
- 示例：Money、Address、DateRange、EmailAddress

### 2.4 领域事件（Domain Event）

- 命名使用过去时态：`OrderCreated`、`PaymentCompleted`、`UserRegistered`
- 事件由聚合根产生，在领域层定义
- 事件处理器放在应用层或基础设施层
- 跨聚合的副作用通过领域事件解耦

### 2.5 领域服务（Domain Service）

- 仅用于不属于任何实体或值对象的领域逻辑
- 无状态，每次调用独立
- 命名体现领域操作：`TransferService`、`PricingCalculator`

### 2.6 仓储（Repository）

- 每个聚合根对应一个仓储接口
- 仓储接口定义在领域层，实现在基础设施层
- 仓储只提供聚合根的持久化操作（增删改查）
- 禁止绕过仓储直接操作数据库

### 2.7 工厂（Factory）

- 复杂对象的创建逻辑封装到工厂中
- 工厂负责保证创建出的对象满足所有不变量
- 简单对象直接用构造函数，不必强行使用工厂

## 三、分层架构

```
Presentation（表现层）
    ↓ 调用
Application（应用层）
    ↓ 编排
Domain（领域层）
    ↑ 实现
Infrastructure（基础设施层）
```

### 3.1 领域层（Domain）

- 纯业务逻辑，不依赖任何框架和基础设施
- 包含：实体、值对象、聚合根、领域事件、领域服务、仓储接口
- 禁止：直接操作数据库、调用外部 API、使用框架注解

### 3.2 应用层（Application）

- 编排领域对象完成用例，不含业务规则
- 包含：应用服务、DTO、事件处理器、事务管理
- 一个应用服务方法 = 一个用例 = 一个事务
- 职责：参数校验 → 调用领域逻辑 → 持久化 → 发布事件

### 3.3 基础设施层（Infrastructure）

- 技术实现细节
- 包含：仓储实现、外部 API 客户端、消息队列、缓存、ORM 配置
- 实现领域层定义的接口（依赖倒置）
- 禁止被领域层或应用层直接依赖具体实现

### 3.4 表现层（Presentation）

- 接收入户请求，转换为应用层调用
- 包含：Controller/Resolver、请求校验、响应序列化
- 禁止包含业务逻辑

---

## 四、依赖方向规则

- 表现层 → 应用层 → 领域层 ← 基础设施层
- 领域层不依赖任何外层
- 跨层引用只能引用接口，不能引用具体实现
- 禁止循环依赖

## 五、目录结构约定

```
src/
├── contexts/                    # 按限界上下文组织
│   ├── ordering/                # 订单上下文
│   │   ├── domain/              # 领域层
│   │   │   ├── aggregates/      # 聚合
│   │   │   ├── entities/        # 实体
│   │   │   ├── valueobjects/    # 值对象
│   │   │   ├── events/          # 领域事件
│   │   │   ├── services/        # 领域服务
│   │   │   └── repositories/    # 仓储接口
│   │   ├── application/         # 应用层
│   │   │   ├── services/        # 应用服务
│   │   │   ├── dto/             # 数据传输对象
│   │   │   └── handlers/        # 事件处理器
│   │   ├── infrastructure/      # 基础设施层
│   │   │   ├── repositories/    # 仓储实现
│   │   │   ├── clients/         # 外部客户端
│   │   │   └── messaging/       # 消息队列
│   │   └── presentation/        # 表现层
│   │       ├── controllers/     # 控制器
│   │       └── dtos/            # 请求/响应 DTO
│   └── billing/                 # 账单上下文
│       └── ...
└── shared/                      # 共享内核
    ├── domain/                  # 通用领域基类
    └── infrastructure/          # 通用基础设施
```

## 六、编码规范

### 6.1 命名

- 聚合根：名词，如 `Order`、`User`、`Product`
- 领域事件：过去时态，如 `OrderCreated`、`PaymentFailed`
- 领域服务：动词+名词，如 `TransferService`、`PricingCalculator`
- 仓储接口：`IOrderRepository`（TypeScript）/ `OrderRepository`（Python 协议类）
- 值对象：名词，如 `Money`、`Address`、`Email`
- 应用服务方法：动词开头，如 `placeOrder()`、`cancelSubscription()`

### 6.2 事务

- 一个事务对应一个聚合根的修改
- 跨聚合操作使用领域事件 + 最终一致性
- 禁止在一个事务中修改多个聚合根

### 6.3 测试

- 领域层：纯单元测试，无外部依赖
- 应用层：集成测试，mock 基础设施
- 测试优先写聚合的行为和不变量

## 七、反模式警告

| 反模式 | 正确做法 |
|--------|---------|
| 贫血模型（实体只有 getter/setter） | 将行为封装在实体/值对象中 |
| 跨聚合直接引用对象 | 只通过聚合根 ID 引用 |
| 领域层依赖 ORM 框架 | 通过仓储接口隔离 |
| 在 Controller 写业务逻辑 | 业务逻辑只在领域层 |
| 一个大聚合包含所有关联数据 | 聚合保持最小，用 ID 引用关联 |
| 跨上下文共享数据库表 | 每个上下文独立数据模型 |
| 应用服务包含业务规则 | 业务规则只在领域层 |
| 值对象可变 | 值对象必须不可变 |
